## 理解令牌

令牌是 AI 交互的货币。您发送的每条消息和 AI 的每个响应都会消耗令牌。

### 什么使用令牌

- **您的提示** - 您发送给 AI 的消息
- **AI 响应** - AI 生成的代码和文本
- **上下文** - 以前的对话历史
- **代码编辑** - 对现有代码的修改

### 什么不使用令牌

- **预览** - 查看您的网站
- **讨论模式** - 在不构建的情况下进行头脑风暴
- **设置更改** - 配置您的项目
- **文件浏览** - 查看项目文件

## 节省令牌的策略

### 使用计划模式

计划模式帮助您在构建之前完善需求：

1. 描述您的完整愿景
2. AI 创建详细计划
3. 审查并调整计划
4. 仅在计划完美时构建

这可以防止因误解需求而导致的昂贵重建。

### 使用讨论模式

在不使用令牌的情况下进行头脑风暴和澄清想法：

```text
[在讨论模式下]
我正在考虑在我的网站上添加博客。
我应该包含哪些区块？组织帖子的最佳方式是什么？
```

一旦您决定了，切换到构建模式并提供清晰的需求。

### 预先要具体

模糊的提示会导致来回纠正：

❌ **浪费：**
```text
您："添加联系表单"
AI：[创建基本表单]
您："添加电话号码字段"
AI：[添加字段]
您："添加验证"
AI：[添加验证]
您："使其成为两列"
AI：[重构表单]
```

✅ **高效：**
```text
您："添加包含姓名、电子邮件、电话和消息字段的联系表单。
使用两列布局。包含所有字段的验证。
在每个字段下方显示错误消息。"
AI：[正确创建完整表单]
```

### 批量相关更改

将类似的更改组合在一起：

❌ **浪费：**
```text
"将按钮颜色更改为蓝色"
"使按钮更大"
"为按钮添加圆角"
"为按钮添加阴影"
```

✅ **高效：**
```text
"更新按钮：蓝色背景、更大尺寸、
圆角和微妙阴影"
```

### 使用可视化编辑器

对于简单的更改，使用可视化编辑器而不是 AI：

- 文本编辑
- 颜色更改
- 图像交换
- 间距调整

为需要代码的复杂更改保存令牌。

### 提供完整信息

在一个提示中包含所有必要的详细信息：

```text
创建包含以下内容的定价区块：
- 三个层级：基础版（$9/月）、专业版（$29/月）、企业版（自定义）
- 每个层级的功能列表
- 专业版层级上的"最受欢迎"徽章
- 年度/月度切换
- 每个层级上的 CTA 按钮
- 响应式设计（在移动设备上堆叠）
```

### 参考现有元素

在现有基础上构建：

```text
"使用与功能区块相同的卡片样式创建推荐区块"
```

这比从头描述样式更清晰、更高效。

## 高效的工作流程

### 从结构开始

首先构建骨架，然后添加细节：

1. **结构：** "创建包含标题、英雄、功能、推荐、定价和页脚的落地页"
2. **内容：** "将此内容添加到英雄区块：[内容]"
3. **样式：** "更新配色方案以使用蓝色和绿色"
4. **润色：** "添加动画和悬停效果"

### 使用模板

从常见模式的模板开始：

```text
"创建标准 SaaS 落地页模板"
```

然后自定义：

```text
"将英雄标题更新为：[您的标题]"
"将功能更改为：[您的功能]"
```

### 战略性迭代

按逻辑顺序进行更改：

1. 正确布局
2. 添加内容
3. 完善样式
4. 添加交互
5. 润色细节

不要随意跳跃——每次迭代都应该建立在前一次的基础上。

### 保存成功的提示

保留一份有效提示的文档：

```text
定价区块提示：
"创建包含三个层级的定价区块。
每个层级应该有：名称、价格、功能列表和 CTA 按钮。
突出显示中间层级。使用带有微妙阴影的卡片设计。"
```

在未来的项目中重复使用这些。

## 高级技术

### 增量构建

分阶段构建复杂功能：

**阶段 1：** 基本功能
```text
"创建显示标题和摘录的博客文章列表"
```

**阶段 2：** 添加功能
```text
"为博客列表添加类别过滤"
```

**阶段 3：** 润色
```text
"为博客添加分页和搜索"
```

### 选择性编辑

编辑特定部分而不是重新生成：

```text
"在英雄区块中，仅将标题更改为：
[新标题]。保持其他所有内容不变。"
```

### 使用代码视图

对于开发者，有时直接编辑代码更快：

1. 使用 AI 生成初始代码
2. 在代码视图中进行小调整
3. 仅对复杂更改使用 AI

### 利用上下文

AI 记住您的对话：

```text
"使用与之前相同的配色方案"
"添加我们之前讨论的功能"
"将相同的样式应用于这个新区块"
```

## 监控令牌使用

### 检查您的余额

在您的账户中监控令牌：

- 查看剩余令牌
- 查看使用历史
- 按项目跟踪令牌

### 设置项目预算

为每个项目分配令牌：

```text
"我有 10,000 个令牌用于此项目。让我们仔细计划
以保持在预算内。"
```

### 构建前估算

询问 AI 估算令牌使用：

```text
"构建包含这些功能的 5 页网站需要多少令牌：
[列出功能]"
```

## 何时自由使用令牌

### 复杂功能

不要在复杂功能上吝啬令牌：

- 数据库集成
- 身份验证系统
- 支付处理
- 复杂交互

这些需要 AI 的全部功能。

### 学习

在学习上投资令牌：

```text
"解释此代码的工作原理并建议改进"
```

理解有助于您以后更高效地构建。

### 关键修复

使用令牌修复重要问题：

```text
"结账流程已损坏。用户无法完成购买。
请调试并修复。"
```

## 令牌节省清单

在发送提示之前，问：

- [ ] 这足够具体吗？
- [ ] 我可以将此与其他更改批量处理吗？
- [ ] 我可以改用可视化编辑器吗？
- [ ] 我是否提供了所有必要的信息？
- [ ] 我是否在适当时使用计划或讨论模式？
- [ ] 这是实现我目标的最有效方式吗？

## 常见的令牌浪费

### 重复纠正

❌ 多次进行相同类型的纠正
✅ 预先提供完整需求

### 模糊请求

❌ "让它看起来更好"
✅ "将标题字体大小增加到 48px，在各区块之间添加 60px 间距"

### 不必要的重新生成

❌ 为小更改重新生成整个区块
✅ 请求对现有代码进行特定编辑

### 不使用讨论模式

❌ 使用构建模式进行头脑风暴
✅ 使用讨论模式进行计划，使用构建模式进行执行

### 忽略可视化编辑器

❌ 使用 AI 进行简单的文本更改
✅ 在可视化编辑器中直接编辑文本
